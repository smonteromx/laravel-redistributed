# Quality Guidelines

Project-specific quality checks and verification expectations.

## Commands

Full quality fixer pipeline:

```bash
composer run quality
```

This is the command that actually formats and fixes both the frontend and backend. It mutates files as needed and runs:
- `pnpm run lint` (`eslint . --fix`)
- `pnpm run format` (`prettier --write resources/`)
- `pnpm run types:check` (`tsc --noEmit`)
- `pint --parallel`
- `phpstan analyse --memory-limit=4G`

Full quality check pipeline:

```bash
composer run quality:check
```

This is the CI-safe, non-mutating quality gate. It is run by `composer run test` and runs:
- `pnpm run lint:check` (`eslint .`)
- `pnpm run format:check` (`prettier --check resources/`)
- `pnpm run types:check` (`tsc --noEmit`)
- `pint --parallel --test`
- `phpstan analyse --memory-limit=4G`

Testing pipeline:

```bash
composer run test
```

This clears config, runs `composer run quality:check`, and runs `php artisan test`.

PHP formatting only:

```bash
vendor/bin/pint --dirty --format agent
```

## Expectations

Rules:
- Never manually edit `AGENTS.md` or `CLAUDE.md`; they are generated by `php artisan boost:update`. Update `.ai/guidelines/*.md` source files instead.
- Run relevant tests after code changes.
- Run `vendor/bin/pint --dirty --format agent` after editing PHP files.
- Run frontend formatting or linting after editing TypeScript, React, CSS, or page/component files.
- Run `composer run quality` (not just `composer run test`) before committing or pushing. `composer run quality` is what mutates and formats both the frontend and backend; `composer run test` only verifies and does not format.
- If a full check is too expensive or blocked, report exactly what ran and what remains unverified.
- Do not create throwaway verification scripts when tests can cover the behavior.
- Use English for code identifiers, filenames, comments, and internal messages; keep user-facing copy in the product language.
- Prefer readable, explicit code over ultra-short or clever code.

## Commits

Use Conventional Commits for every commit in this project.

Rules:
- Prefer a scoped subject whenever a clear scope exists: `type(scope): summary`.
- Keep commit messages to a single subject line; do not add commit bodies.
- Do not mention AI agents, assistants, tools, or co-authors anywhere in the commit message.
- Use the type that best describes the intent, such as `feat`, `fix`, `refactor`, `test`, `docs`, `style`, `chore`, or `build`.
- Keep the summary imperative and focused on why the change exists.

Examples:
- `feat(auth): add password reset flow`
- `fix(users): prevent duplicate email validation conflicts`
- `docs(guidelines): split project conventions by scope`
